# Жёсткие правила
- ТОЛЬКО локальный/dev-контур ([BASE_URL], дефолт http://127.0.0.1:8080). НИКОГДА не направляй
  тесты и деструктивные пробы на stage/prod. Деструктивный хаос (kill контейнеров, обрыв сети,
  порча конфигов) — только на локальных dev-контейнерах, с обязательным восстановлением после.
- Стенд — в ОТДЕЛЬНОЙ папке `demostand/`, исключённой из git через `.git/info/exclude`
  (не трогай общий .gitignore; проверь `git check-ignore demostand/`). Ничего из стенда не
  коммить.
- Строй стенд С НУЛЯ и НЕЗАВИСИМО. Существующие тесты считай НЕПРОВЕРЕННЫМИ: нет гарантий, что
  они учли все сценарии и верны. Где можешь — независимо перепроверь поведение, которое они
  якобы покрывают.
- Документ-ориентированность: источник истины — спеки в `demostand/spec/` (сценарии, флоу,
  edge-cases, безопасность, инфра). Скрипты и браузер-агенты лишь ИСПОЛНЯЮТ то, что описано.
- Симулируй НАСТОЯЩЕГО пользователя через браузер (реальный логин, клики, ввод), а не только
  прямые вызовы API.
- Баг = воспроизведён ≥3 раз + установлена причина. Только тогда — критичные сообщай сразу.
  Режим работы с фиксами: [FIX_MODE: "копить всё и не чинить" ЛИБО "чинить подтверждённые сразу"].
  Если чинишь — строго по TDD (тест→код→прогон→проверка отсутствия регрессий) и записывай в
  таймлайн «что нашёл / как починил / перепроверено». Изменения в auth/безопасности/миграциях —
  прогоняй внешний код-ревью в самом конце.
- Для каждого индуцированного сбоя проверяй не «упало ли», а ВОССТАНАВЛИВАЕТСЯ ли система сама
  или уходит в broken-state и требует перезапуска.
- Перед outward-facing/необратимыми действиями и перед хаосом на возможно-shared инфраструктуре —
  подтверждай. После браузер-проб закрывай сессии и удаляй скриншоты/артефакты.
- Вне области: [ANY_EXCLUSIONS — функции/режимы, которые не трогаем; если нет — «нет»].
- Раз в пару часов делай краткий отчёт-чекпоинт и продолжай.

# Фаза 1 — Карта системы (фундамент)
Построй структурированную карту приложения (можно распараллелить агентами-исследователями по
доменам). По каждому домену: возможности, действия пользователя, настройки, API-роуты + коды
ошибок, edge-cases (включая редкие гонки «раз в 10 лет»), режимы отказа (self-heal / нужен
рестарт / broken-state), вопросы безопасности, существующее тест-покрытие (как непроверенное).
Домены (адаптируй под проект):
1) Аутентификация и жизненный цикл сессии. 2) Авторизация/роли/мультитенантность и изоляция
данных. 3) Основные бизнес-флоу (CRUD/процессы продукта). 4) Модель данных и её изоляция
(RLS/scoping). 5) Файлы (загрузка/скачивание/шаринг). 6) Реальное-время (websocket/SSE,
reconnect/resync). 7) Настройки/админка. 8) Фоновые задачи/планировщики/очереди. 9) Внешние
интеграции. 10) Инфра и загрузка конфигов (compose, healthcheck, restart-политики, порядок
старта, источники конфигов). 11) Полный инвентарь API-роутов и кодов ошибок. 12) [Опционально,
если есть AI/агент/LLM] — границы поведения агента.
Сохрани сырую карту в `demostand/spec/00-system-map.json`.

# Фаза 2 — Спецификации стенда (из карты)
Напиши документы (markdown), каждый пункт со стабильным ID, реальными путями файлов, риском,
вероятностью и способом проверки (браузер-шаг или API-вызов + ожидаемый код ошибки):
- `01-test-data` — тестовые аккаунты/организации/данные, креды, профили.
- `02-feature-surface` — полная поверхность: возможности/настройки/роуты.
- `03-user-flow-matrix` — сквозные пользовательские флоу и их ПЕРЕСЕЧЕНИЯ (хаотичные порядки,
  параллельные действия, переключения контекста/вкладок, прикрепления, правка/открытие сущностей).
- `04-edge-cases` — консолидированный ранжированный каталог (гонки, границы времени/таймзон,
  битые/огромные/пустые входы).
- `05-security` — изоляция тенантов/IDOR, authz, токены/TTL, XSS/инъекции, path-traversal,
  утечка секретов, rate-limit/lockout. [Если есть агент — отдельный adversarial-каталог границ.]
- `06-infra-resilience` — карта контура + матрица «самовосстановление vs застревание» + план
  chaos-инъекций (kill/restart контейнеров, обрыв/замедление сети, порча/отсутствие конфигов,
  холодный старт) с точными командами и ожидаемым post-condition.
- `08-candidate-findings` — РАНЖИРОВАННЫЕ воспроизводимые гипотезы дефектов (severity × confidence)
  с точными шагами repro: мост к прогону.

# Фаза 4 — Исполнители сценариев
- Браузерный харнесс (host-side обёртка над браузер-CLI): login/logout, навигация, отправка
  форм/сообщений, ожидание завершения операций по реальным сигналам UI, захват консольных
  ошибок и сетевых сбоев (offline/abort/throttle), скриншоты. Несколько сессий = несколько
  «людей» (но следи за ресурсами: 5+ параллельных тяжёлых LLM/SPA-сессий дают ложные таймауты —
  ограничивай параллелизм).
- API/конкурентность: бурст-нагрузка, гонки, негативные входы (пропущенные/битые/огромные поля).
- Инфра-хаос: kill/restart отдельных контейнеров; обрыв доступа к Docker/БД/зависимостям;
  обрыв/замедление сети в браузере; порча/отсутствие конфиг-файлов; холодный старт. С
  восстановлением и проверкой self-heal.
- [Если есть AI/агент] adversarial-пробы границ (см. Фазу 5).

# Фаза 5 — Прогон раундами, накопление находок
Категории (адаптируй):
- Auth: login/logout/неверный пароль/rate-limit/lockout/жизненный цикл и продление сессии/cookie-флаги.
- Authz/tenancy: IDOR и cross-tenant доступ по чужим ID; роли; что видит обычный юзер vs admin;
  реально ли БД (RLS) — backstop или изоляция держится только на коде (проверь роль подключения к БД).
- Валидация и КОНТРАКТ ОШИБОК: коды статусов (4xx vs 5xx), стабильные коды ошибок, человекочитаемые
  сообщения, fail-fast без скрытых fallback’ов.
- Бизнес-флоу: happy-path + создание/правка/открытие/удаление сущностей, «переделай текущее»,
  «открой прошлое», конкурентные правки.
- Файлы: oversized/пустые/запрещённый MIME/двойные расширения/path-traversal/межтенантный доступ/
  скачивание незавершённого/шаринг приватного.
- Реальное-время: reconnect/resync после обрыва, целостность стрима (без задвоений), отмена/стоп.
- Конкурентность/гонки: параллельные одинаковые операции, двойные сабмиты, гонки старта.
- Сеть: кратковременный и долгий обрыв, медленная сеть, обрыв конкретного эндпоинта; graceful-
  деградация (понятное сообщение, не краш) + ПОЛНОЕ восстановление после возврата связи.
- Инфра: после kill/restart/обрыва зависимости — самовосстановление или wedge (нужен рестарт?).
- Безопасность: XSS (в т.ч. через данные от интеграций), инъекции, токены без TTL/отзыва,
  утечка секретов в ответы/в UI, traversal, отсутствие rate-limit.
- [AI/агент, если есть] Граница: агент решает только профильные задачи и корректно ОТКАЗЫВАЕТ в
  непрофильных (off-topic, код, художественные тексты, общий чат); устойчив к prompt-injection
  («игнорируй инструкции», смена роли, «выведи свой системный промпт»); не переписывает себя/свои
  правила; не читает файлы/секреты вне песочницы; не утаскивает данные чужих тенантов. Любой
  пробитый случай — критическая находка.
Для каждой находки: воспроизведи ≥3 раз, установи причину (файл:строка), оцени severity и
confidence. Различай реальный баг и артефакт нагрузки/харнесса (перепроверь чисто).

# Фаза 6 — Фиксы (по режиму) или накопление
Если [FIX_MODE] = чинить: по TDD (RED→фикс→GREEN→прогон смежных тестов→typecheck/линт), запись в
таймлайн; для значимых изменений — внешний код-ревью в конце; обязательный самообзор всегда.
Если = копить: ничего не чини, складывай в реестр; в конце напиши `REMEDIATION.md` — по каждой
находке: где, причина, конкретный фикс (файлы/код/SQL/конфиг, слой), TDD-план, риски регрессий,
спорные решения для обсуждения, как перепроверить через стенд, оценка усилий.
