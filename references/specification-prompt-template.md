Ты выступаешь как главный архитектор продукта, системный аналитик и технический руководитель. Твоя задача — подготовить не обзор и не рекламное описание, а каноническую спецификацию реализации, по которой команда и AI-агенты смогут строить продукт без постоянных устных уточнений.

Пиши спецификацию на русском языке, обычным понятным языком, но с достаточной технической точностью. Если используешь технический термин, сразу объясняй его простыми словами. Не пиши общие фразы вроде "нужно продумать безопасность". Вместо этого фиксируй конкретные правила, таблицы, состояния, ограничения, проверки и критерии готовности.

## Главная Цель

Создай спецификацию такого уровня, чтобы по ней можно было:

- проектировать базу данных;
- строить серверную часть;
- строить пользовательские интерфейсы;
- писать интеграции;
- писать тесты;
- проверять безопасность;
- разворачивать продукт;
- поручать задачи AI-агентам без потери контекста.

Документ должен быть строгим: если поведение не описано, реализация не должна его молча придумывать.

## Обязательные Принципы Документа

- Сначала зафиксируй границы продукта: что входит сейчас, что не входит сейчас, что оставляется на будущее.
- Разделяй пользовательские сценарии, бизнес-сущности, технические каналы, уведомления, аналитику и внешние интеграции.
- Для каждой важной операции укажи действующее лицо, условия запуска, успешный результат и побочные действия.
- Для сущностей со статусами опиши конечный автомат: откуда куда можно переходить, кто может это делать, какие инварианты нельзя нарушать.
- Для данных укажи, что является источником истины, что является производной витриной, что хранится неизменно, а что можно менять.
- Для прав доступа используй правило "запрещено по умолчанию".
- Не придумывай скрытые значения по умолчанию для обязательных данных и настроек. Если значение нужно для правильной работы, система должна явно требовать его и падать с понятной ошибкой при отсутствии.
- AI должен быть советником, а не невидимым владельцем бизнес-состояния: все AI-результаты проходят проверку схемы, порог уверенности и детерминированные правила.
- Все внешние побочные действия должны быть явными: события, фоновые задачи, журнал действий, повторные попытки, подавление дублей.
- Ошибки должны иметь стабильный код, понятное человеку сообщение и подсказку следующего шага там, где это полезно.
- Архитектурные правила должны быть проверяемыми автоматикой там, где это возможно.

## Требуемая Структура Спецификации

Сформируй документ со следующими разделами.

### 1. Product Scope

Опиши:

- назначение продукта;
- текущий этап реализации;
- что входит в этап;
- что не входит;
- какие будущие этапы учитываются только как точки расширения;
- правило, запрещающее превращать текущий этап в одноразовый прототип, если продукт должен жить долго.

Добавь подраздел `Scope Rules` с жесткими правилами изменения границ.

### 2. Glossary

Сделай таблицу терминов.

Для каждого термина укажи:

- название;
- простое определение;
- важные примечания;
- чем этот термин не является, если есть риск спутать его с другим.

После таблицы добавь `Glossary Rules`: список запретов на смешивание похожих понятий.

### 3. Universal Core And Product Templates

Если продукт может использоваться в разных отраслях, ролях, странах, типах клиентов или бизнес-моделях, опиши:

- что является универсальным ядром;
- что задается настройками или шаблонами;
- какие отличия не должны плодить отдельные параллельные реализации;
- какие стартовые шаблоны нужны сейчас;
- какой следующий шаблон или сценарий должен быть возможен без переписывания ядра.

Если универсальность не нужна, явно объясни почему и какие границы все равно нужно не смешивать.

### 4. Use-Case Catalog

Составь каталог ключевых сценариев.

Разбей его по областям продукта.

Каждая строка должна содержать:

- `ID`;
- название сценария;
- основное действующее лицо;
- предварительные условия;
- успешный результат;
- основные побочные действия.

Обязательно покрой:

- вход и управление пользователями;
- основные рабочие сценарии продукта;
- создание, изменение и закрытие ключевых бизнес-объектов;
- внешние интеграции;
- фоновые задачи;
- уведомления;
- аналитику;
- настройки клиента или администратора;
- публичные поверхности, если они есть.

Добавь `Use-Case Rules`:

- каждая команда, обработчик, экран, фоновая задача и AI-сценарий должны ссылаться на один или несколько use case ID;
- скрытые побочные действия запрещены;
- если новая функция не маппится на каталог, сначала обновляется спецификация.

### 5. Canonical State Machines

Для каждой сущности со статусами опиши таблицу:

- `From`;
- `To`;
- `Trigger`;
- `Allowed actors`;
- `Notes`.

После каждой таблицы добавь инварианты:

- какие состояния конечные;
- какие переходы запрещены;
- какие переходы требуют причины;
- что пишется в историю;
- какие прямые изменения статуса запрещены.

Минимально рассмотри:

- основную рабочую сущность продукта;
- назначение или владение задачей, если есть;
- пользовательскую сессию;
- импорт или массовую операцию, если есть;
- уведомление или доставку, если есть;
- публикацию контента, если есть.

### 6. Canonical Event Catalog

Опиши события как факты, которые уже произошли, а не как команды.

Сначала зафиксируй правила:

- где события создаются;
- как обеспечивается повторная доставка;
- как потребители должны быть устойчивы к повторной обработке;
- как версионируются события;
- почему события не заменяют основную базу данных, если это не event sourcing.

Опиши конверт события:

- event id;
- name;
- version;
- time;
- tenant/client/account id, если продукт многоклиентский;
- aggregate type;
- aggregate id;
- actor;
- correlation id;
- causation id;
- payload.

Сделай таблицы событий по областям продукта:

- событие;
- кто публикует;
- когда публикуется;
- обязательные поля;
- типичные потребители.

### 7. Data Model Level 2

Опиши логическую модель данных на уровне таблиц или коллекций.

Для каждой таблицы укажи:

- название;
- назначение;
- обязательные поля;
- ключевые ограничения.

Обязательно опиши:

- идентификаторы;
- даты и время;
- мягкое архивирование или удаление;
- неизменяемые записи;
- историю действий;
- связи между сущностями;
- правила для JSON/гибких полей;
- правила для персональных и чувствительных данных.

Если есть многоклиентность, укажи:

- где хранится `tenant/account/org id`;
- как запрещаются пересечения данных;
- какие таблицы могут быть глобальными;
- как строятся уникальные ключи.

### 8. API Contract Baseline

Опиши выбранный стиль API.

Укажи:

- базовый префикс;
- формат данных;
- источник истины для схем;
- как генерируется документация;
- где не должна жить бизнес-логика.

Сделай каталог точек доступа в таблицах:

- метод;
- путь;
- кто имеет доступ;
- связанные use case ID;
- нужна ли защита от повторной отправки запроса;
- примечания.

Добавь единый формат ошибки:

json
{
  "error": {
    "code": "STABLE_ERROR_CODE",
    "message": "Понятное человеку объяснение, что пошло не так.",
    "hint": "Что сделать дальше, если это применимо.",
    "requestId": "...",
    "details": {}
  }
}

Опиши правила:

- когда обязателен ключ защиты от повторной отправки;
- как устроена пагинация;
- как задаются фильтры;
- как обрабатываются несовместимые изменения;
- как проверяются внешние webhooks/callbacks;
- почему DTO не должны протекать напрямую из базы без явной схемы.

### 9. Canonical Architecture

Опиши архитектурный baseline:

- тип репозитория;
- стиль модульности;
- границы приложений;
- где живет бизнес-логика;
- где живут интеграции;
- где живут фоновые задачи;
- где живет пользовательский интерфейс;
- где живет общая техническая платформа.

Сделай список выбранных технологий и список явно запрещенных baseline-решений, если они создают риск сложности или скрытой логики.

### 10. Source-Of-Truth Rules

Опиши, что является главным источником истины для:

- продуктовых правил;
- бизнес-логики;
- API-схем;
- схемы базы данных;
- миграций;
- настроек времени выполнения;
- маршрутизации событий;
- пользовательских интерфейсов.

Отдельно перечисли, где запрещено хранить критичную бизнес-логику: например, только в подсказках AI, только в SQL-витринах, только в интерфейсе, только в CI-скриптах или во внешнем конструкторе процессов.

### 11. Repo Layout

Предложи структуру репозитория.

Опиши:

- приложения;
- модули по бизнес-областям;
- слой контрактов;
- доменный слой;
- функциональные вертикальные срезы;
- инфраструктурные адаптеры;
- тесты;
- платформенные возможности;
- документацию;
- инструменты проверки архитектуры.

После дерева папок добавь правила:

- какие импорты разрешены;
- какие импорты запрещены;
- где должны лежать тесты;
- где должна находиться публичная граница модуля;
- какие папки запрещены, чтобы не появлялись бесформенные `utils/common/helpers`;
- какие архитектурные правила должны проверяться автоматически.

### 12. AI-Agent Delivery Rules

Опиши правила, чтобы AI-агенты могли безопасно развивать кодовую базу:

- перед изменением читать спецификацию и локальные контракты;
- всегда определять bounded context и use case;
- тесты перед логикой для изменения поведения;
- минимальное корректное изменение;
- запрет скрытых значений по умолчанию для обязательных данных;
- понятные ошибки;
- ограничение размера файлов;
- комментарии к неочевидным логическим блокам;
- внешние побочные действия только через явные адаптеры, события или фоновые задачи;
- автоматическая проверка архитектуры как обязательный барьер.

### 13. Bounded Contexts

Сделай таблицу бизнес-областей:

- контекст;
- чем владеет;
- чем не владеет.

Важно: явно отделяй похожие области, которые легко смешать. Например: коммуникации и уведомления, пользователь и сессия, платеж и счет, заказ и сообщение, файл и вложение, событие и команда.

### 14. Canonical Domain Model

Сделай таблицу агрегатов и ключевых сущностей:

- сущность;
- назначение;
- инварианты.

После таблицы добавь правила моделирования:

- какие сущности не равны друг другу;
- какие связи допускаются;
- где нужна история;
- где нельзя физически удалять;
- какие AI-результаты не являются состоянием до проверки;
- какие аналитические данные являются производными.

### 15. Multitenancy Or Ownership Model

Если продукт многоклиентский, опиши модель изоляции клиентов.

Если продукт не многоклиентский, опиши модель владения данными пользователями, организациями, командами или проектами.

Укажи:

- как определяется владелец данных;
- какие данные глобальные;
- как проверяется доступ;
- как строятся ключи;
- как фоновые задачи сохраняют контекст владельца;
- как файлы и аналитика сохраняют границы доступа.

### 16. Data Rules And Import Rules

Опиши общие правила данных:

- формат идентификаторов;
- время и часовые пояса;
- архивирование;
- история изменений;
- персональные данные;
- JSON-поля;
- внешние ключи;
- правила именования таблиц.

Если есть импорт, массовое обновление или синхронизация, опиши:

- двухшаговый процесс `preview/validation` -> `execute`;
- ошибки по строкам;
- явное подтверждение опасных изменений;
- запрет неявных удалений;
- политику слияния;
- коды ошибок;
- контракт ответа preview.

### 17. Auth, Sessions And CSRF

Опиши:

- кто является источником авторизации;
- типы входа;
- сессии;
- сроки жизни;
- rotation/revocation;
- защита браузерных запросов;

### 18. RBAC Matrix

Сделай матрицу ролей:

- роль;
- область действия;
- основные права.

Затем сделай таблицу прав:

- код права;
- смысл;
- допустимые роли;
- ограничения области.

Правила:

- deny-by-default;
- все права проверяются на сервере;
- привилегированные действия логируются;
- если права нет в спецификации, оно запрещено.

### 19. Channel And Integration Model

Если продукт принимает данные из внешних каналов или отправляет наружу, опиши:

- каналы;
- модель авторизации каждого канала;
- как канал связывается с клиентом/пользователем/организацией;
- что считается достоверной личностью;
- как обрабатываются вложения;
- как обрабатываются повторы;
- как хранятся исходные внешние данные;
- какие действия канал не имеет права выполнять напрямую.

Для каждого канала сделай контракт:

- направление;
- привязка;
- источник личности;
- идентификатор диалога или операции;
- идентификатор сообщения или события;
- вложения;
- ответы;
- подтверждения доставки;
- отказоустойчивость;
- специальные правила.

### 20. Frontend Runtime And UI Flow Specs

Опиши все пользовательские поверхности.

Для каждой поверхности сделай таблицу сценариев:

- `Flow ID`;
- экран или точка входа;
- основное действующее лицо;
- счастливый путь;
- обязательные состояния ошибки, пустого результата, загрузки, запрета доступа и конфликта.

Добавь правила интерфейса:

- интерфейс не рассчитывает права как источник истины;
- устаревшее состояние не перезаписывается молча;
- опасные действия требуют подтверждения;
- ошибки должны быть понятны пользователю;
- важные статусы, история и причины решений должны быть видимы.

### 21. Main Business Lifecycle

Опиши жизненный цикл главной бизнес-сущности продукта обычным языком и таблицей.

Укажи:

- внутренние статусы;
- пользовательские статусы, если они отличаются;
- кто может менять статус;
- что происходит с историей;

### 22. Assignment, Routing, SLA Or Workload Rules

Если в продукте есть назначение работы, маршрутизация, очереди, сроки или нагрузка, опиши:

- как выбирается исполнитель или очередь;
- какие данные используются;
- что запрещено выбирать;
- как считаются сроки;
- что ставит срок на паузу;
- что считается нарушением;
- как обрабатываются гонки и одновременные изменения;
- какие изменения должны быть видны в истории.

### 23. Dedupe, Merge And Conflict Rules

Если возможны дубли или слияние данных, опиши:

- критерии дубликата;
- что может предложить AI;
- что должно подтвердить правило или человек;
- какие слияния запрещены;
- как хранится связь с исходными объектами;
- как откатывается ошибочное решение;
- какие конфликты блокируют операцию.

### 24. AI Operating Model

Опиши все AI-сценарии.

Для каждого укажи:

- use case;
- входные данные;
- выходную схему;
- порог уверенности;
- что делает система при низкой уверенности;
- какие поля AI не имеет права менять;
- какие проверки выполняются после ответа;
- как логируются запросы;
- как устроены оценки качества;
- что считается недопустимым ответом.

Правила:

- AI не расширяет права доступа;
- AI не обходит согласия, сроки, настройки и политики;
- AI не создает обязательные данные из воздуха;
- AI-ответ не становится бизнес-состоянием без проверки.

### 25. Files And Media

Если продукт работает с файлами, опиши:

- где хранятся байты;
- где хранится метаинформация;
- как создаются ссылки на бизнес-объекты;
- как проверяется доступ;
- как выполняется сканирование;
- как выдаются временные ссылки;
- что логируется;
- какие типы файлов разрешены;
- что происходит при блокировке файла.

### 26. Notifications And Campaigns

Если продукт отправляет уведомления, опиши:

- чем уведомление отличается от обычного сообщения или комментария;
- типы уведомлений;
- аудиторию;
- согласия и предпочтения;
- выбор канала;
- тихие часы;
- подавление дублей;
- повторные попытки;
- причины окончательного отказа;
- двухшаговый процесс для массовых отправок: preview -> publish/schedule;
- контракт preview;
- коды подавления.

### 27. Analytics And Read Models

Опиши аналитику как отдельный слой, а не как тяжелые запросы напрямую к рабочим таблицам.

Укажи:

- источники данных;
- витрины или read models;
- частоту обновления;
- показатели;
- формулы;
- что видит пользователь при устаревших данных;
- какие виджеты могут падать независимо;
- как строятся drill-down переходы;
- какие выгрузки разрешены.

### 28. Observability

Опиши обязательные поля логов:

- request id;
- trace id;
- owner/tenant/account id;
- actor id;
- module;
- use case;
- entity type;
- entity id;
- provider, если применимо.

Перечисли, что обязательно логировать.

Опиши базовые сигналы тревоги:

- всплеск ошибок;
- задержка очередей;
- падение внешнего поставщика;
- ошибки AI;
- ошибки файлов;
- проблемы базы;
- проблемы доставки уведомлений.

### 29. Security, Privacy And Compliance

Опиши:

- шифрование;
- секреты;
- минимальные права;
- аудит;
- персональные данные;
- бэкапы;
- требования закона или отрасли;
- запрет секретов в репозитории;
- запрет токенов в логах;
- доступ поддержки;
- подтверждение разрушительных действий;
- сроки хранения данных.

### 30. Runtime, Staging And Deploy

Опиши:

- состав сервисов;
- окружения;
- reverse proxy или входную точку;
- сборку;
- registry;
- миграции;
- порядок deploy;
- readiness/liveness/startup проверки;
- smoke checks;
- rollback;
- запрет ручных hotfix внутри работающих контейнеров или серверов, если это важно для проекта.

### 31. Environment And Config Contract

Опиши классы настроек:

- база данных;
- кэш;
- файлы;
- авторизация;
- интеграции;
- AI;
- наблюдаемость;
- публичные настройки интерфейса.

Для каждого класса укажи:

- примеры;
- что происходит при отсутствии;
- может ли сервис стартовать в ограниченном режиме;
- где происходит проверка схемы.

Главное правило: обязательная настройка не получает выдуманное значение по умолчанию.

### 32. Backup And Disaster Recovery

Опиши:

- что бэкапится;
- расписание;
- как проверяется восстановление;
- порядок восстановления;
- что считается неполным восстановлением;
- как часто проводятся учебные восстановления.

### 33. Testing Strategy

Опиши risk-based testing: тестировать сначала критичные риски, а не косметику.

Сделай:

- список видов тестов;
- список критичных сквозных сценариев;
- матрицу минимального покрытия по модулям;
- матрицу обязательных тестов по типу изменения;
- release gate matrix.

Обязательно включи:

- тесты доменных правил;
- интеграционные тесты базы и фоновых задач;
- контрактные тесты API;
- тесты прав доступа;
- тесты миграций;
- тесты повторной доставки для фоновых обработчиков;
- AI eval suite, если есть AI;
- regression tests для нетривиальных исправлений.

### 34. Non-Functional Requirements

Опиши:

- доступность;
- p95 скорости чтения;
- p95 скорости изменений;
- задержку фоновых задач;
- ограничения на тяжелые операции;
- масштабирование;
- деградацию при отказе внешних поставщиков.

Не используй абстрактные обещания. Дай численные ориентиры или явно укажи, что их нужно подтвердить нагрузочным тестом.

### 35. Feature Delivery Gates

Опиши `Definition of Ready`:

- есть владелец области;
- есть use case;
- описаны действующее лицо, запуск, счастливый путь, ошибки;
- описаны входы и выходы;
- описаны изменения данных;
- описаны права и аудит;
- описаны тесты и критерии приемки.

Опиши `Definition of Done`:

- тесты пройдены;
- проверки типов и качества пройдены;
- наблюдаемость добавлена;
- спецификация обновлена при изменении границ;
- нет скрытых fallback;
- ошибки понятные;
- права и границы данных проверены;
- понятен rollout и rollback.

### 36. Canonical Implementation Decisions

Список решений, которые больше не являются открытыми вопросами.

Для каждого решения укажи:

- что выбрано;
- что отвергнуто;
- почему;
- какое последствие для реализации.

### 37. Deferred Alignment Items

Список вопросов, которые можно отложить, но нельзя забыть.

Для каждого укажи:

- что еще не решено;
- когда нужно решить;
- какие границы уже зафиксированы;
- что запрещено делать до решения.

### 38. Final Rule

Заверши документ правилом:

Если быстрая реализация нарушает эту спецификацию, нужно исправлять реализацию, а не размывать спецификацию ради удобства.

## Проверка Качества Перед Ответом

Перед тем как выдать финальную спецификацию, проверь себя по чек-листу:

- Есть явные границы текущего этапа и будущих этапов.
- Есть словарь терминов и запреты на смешивание похожих понятий.
- Есть каталог use case с ID.
- Есть конечные автоматы для сущностей со статусами.
- Есть события и правила повторной обработки.
- Есть модель данных с обязательными полями и ограничениями.
- Есть API-каталог и единый формат ошибок.
- Есть права доступа и правило deny-by-default.
- Есть правила для AI, если AI используется.
- Есть правила интеграций и каналов, если они есть.
- Есть UI-сценарии с ошибочными и пустыми состояниями.
- Есть аналитика как отдельный слой, если нужна аналитика.
- Есть наблюдаемость, безопасность, хранение данных, deploy, бэкапы и восстановление.
- Есть стратегия тестирования и критерии готовности.
- Нет скрытых значений по умолчанию для обязательных данных.
- Нет фраз-заглушек без конкретного решения.

Если входных данных недостаточно для точного решения, не выдумывай критичные детали. Сделай два блока:

1. `Assumptions` — явно помеченные рабочие допущения.
2. `Open Questions` — вопросы, без которых нельзя принимать окончательное архитектурное решение.

Но даже при открытых вопросах подготовь максимально полную черновую спецификацию, явно отделив твердые решения от допущений.
